1. Prehlad projektu

GUIVideoProcessing je Windows desktop aplikacia (.NET 8, WinForms), ktora v realnom case snima MJPEG stream z ESP32-CAM modulu, spracovava obraz a rozpoznava dvojciferne cislo zobrazene na 7-segmentovom LCD displeji.

1.1 Co aplikacia robi

  1. Pripoji sa na MJPEG stream z ESP32-CAM cez HTTP
  2. Z kazdeho frame vyreze ROI (Region of Interest) – oblast s LCD displejom
  3. Zvacsi ROI, aplikuje image processing pipeline (bilateral filter, CLAHE, threshold, morphology)
  4. Rozpozna dve cifry (lavu a pravu) pomocou ONNX neuronovej siete alebo 7-SEG algoritmu
  5. Vysledky zobrazuje v UI, zapisuje do log suboru (res.txt) a volitelne do MySQL databazy
  6. Vyhodnocovanie prebieha periodicky (default kazdych 10 sekund)

5. Image Processing Pipeline

Metoda Pipeline() v MainForm.cs spracovava kazdy frame v 8 krokoch. Vstupom je _lastRoi (ROI vyrez z frame), vystupom su rozpoznane cifry. Vsetky parametre su citane z UI kontroliek v realnom case (thread-safe cez GetNudValueSafe).

Krok 1: Grayscale konverzia
Cv2.CvtColor(upCopy, gray, BGR2GRAY)
Prevedie farebny BGR obraz na 1-kanalovy grayscale.
Krok 2: Bilateral Filter
Cv2.BilateralFilter(gray, smooth, d, sigmaColor, sigmaSpace)
Edge-preserving smoothing – znizi sum pri zachovani ostrych hran cislic. Parametre: BilateralD=5, SigmaColor=40, SigmaSpace=40.
Krok 3: CLAHE (Contrast Limited Adaptive Histogram Equalization)
clahe.Apply(smooth, contrast)
Lokalne zvysenie kontrastu – zvyrazni biele cislice na zelenom/ciernom pozadi. Parametre: ClipLimit=5.0, TileGrid=2x2.
Krok 4: Adaptive Threshold
Cv2.AdaptiveThreshold(contrast, binary, 255, GaussianC, BinaryInv, blockSize, c)
Binarizacia – kazdy pixel je biely (255) alebo cierny (0). BinaryInv = cislice biele, pozadie cierne. Parametre: BlockSize=11, C=2.0.
Krok 5: Morphology (Open + Close)
Cv2.MorphologyEx(binary, binary, Open/Close, kernel, iterations)
Open = erozia + dilatacia (odstrani male bodky). Close = dilatacia + erozia (vyplni male diery). Parametre: KernelSize=3, OpenIter=1, CloseIter=1.
Krok 6: Connected Components
Cv2.ConnectedComponentsWithStats(binary, labels, stats, centroids)
Najde spojene komponenty a odstrani objekty mensie nez MinArea=30 px². Vysledok: cleaned – binarny obraz len s velkymi objektami (cislicami).
Krok 7: Rozdelenie na 2 cifry
Cleaned obraz sa rozdeli na lavu cast (FromLeft=50% sirky) a pravu cast (FromRight=50% sirky). Kazda cast obsahuje jednu cifru.
Krok 8: Rozpoznavanie cislic
Podla vybranej metody (ONNX alebo 7-SEG) sa kazda cast rozpozna ako cifra 0-9. Vysledok: dvojciferne cislo (napr. "70") alebo "??" ak zlyhalo.

6.1 ONNX neuronova siet (default)

Trieda: OnnxDigitRecognizer (OnnxDigitRecognizer.cs)

Model metadata

ParameterHodnota
Subormodel/model.onnx
ProducerPyTorch 2.8.0
Input"input" – FLOAT [batch, 1, 98, 64] (grayscale, vyska 98, sirka 64)
Output"logits" – FLOAT [batch, 10] (logity pre cifry 0-9)

Inference pipeline

  1. Grayscale konverzia – ak je vstup farebny, konvertuje sa na 1-kanalovy
  2. Resize na 64×98 – InterpolationFlags.Area (downsampling)
  3. Normalizacia – pixely 0-255 → 0.0-1.0
  4. Tensor – DenseTensor<float> shape [1, 1, 98, 64]
  5. Inference – _session.Run(inputs)
  6. Softmax – logity → pravdepodobnosti (numericky stabilny: odcitanie max)
  7. Vysledok – (int digit, float confidence)

Cifra je akceptovana len ak confidence ≥ OnnxMinConfidence (default 0.5). Inak je vysledok null ("?").

public (int digit, float confidence) RecognizeDigit(Mat digitMat)
{
    using var gray = new Mat();
    // ... grayscale konverzia ...

    using var resized = new Mat();
    Cv2.Resize(gray, resized, new Size(64, 98));

    // Normalizacia + tensor [1, 1, 98, 64]
    var inputData = new float[98 * 64];
    for (int y = 0; y < 98; y++)
        for (int x = 0; x < 64; x++)
            inputData[y * 64 + x] = resized.At<byte>(y, x) / 255f;

    var tensor = new DenseTensor<float>(inputData, new[] { 1, 1, 98, 64 });

    using var results = _session.Run(inputs);
    var logits = results.First().AsEnumerable<float>().ToArray();
    var probs = Softmax(logits);

    return (predictedDigit, maxProbability);
}

7. Stream Capture – HttpClient MJPEG

Architekturalna zmena (2026-02-22): Predchadzajuci pristup pouzival VideoCapture (OpenCV/C++ wrapper), ktory nemal timeout pre TCP spojenie a pri uplnom vypadku kamery sa zasekol indefinitne – az na casy dlhsie ako 6 minut (zachytene v logu). Nova implementacia pouziva HttpClient (.NET managed) s explicitnymi CancellationToken timeoutmi a deterministickym spravanim pri vsetkych chybovych stavoch.

7.1 Princip MJPEG streamu

ESP32-CAM posiela MJPEG stream ako nekonecnu HTTP odpoved formatu multipart/x-mixed-replace. Server otvori TCP spojenie a kontinualne posiela jednotlive JPEG ramce oddelene --boundary reazcom: 

HTTP/1.1 200 OK
Content-Type: multipart/x-mixed-replace; boundary=frame

--frame


Content-Type: image/jpeg


Content-Length: 8432





[8432 bajtov JPEG dat]
--frame


Content-Type: image/jpeg


...

Namiesto parsovania HTTP hlaviciek aplikacia hlada JPEG hranice priamo v byte streame pomocou JPEG markerov – standardizovane hodnoty definovane v JFIF/JPEG specifikacii:

  • JPEG zaciatok (SOI): sekvencia bajtov 0xFF 0xD8 0xFF
  • JPEG koniec (EOI): sekvencia bajtov 0xFF 0xD9

Tento pristup je robustnejsi ako parsovanie Content-Length – funguje aj ked kamera vynecha alebo zmeni format hlaviciek, a nevyzaduje implementaciu HTTP chunked parsera.

7.2 Architektura CaptureLoop – dve slucky

Metoda CaptureLoop(CancellationToken) bezi na background thread-e (cez Task.Run) a ma dvojurovnovu strukturu: 

VONKAJSIA SLUCKA (reconnect loop) – bezi donekonecna, stopne ju len STOP
  |
  +-- Backoff sleep: min(pokus * 1s, MjpegReconnectBackoffMaxSec)
  |   pokus 1=1s, 2=2s, ... 60=60s, 61=60s, 62=60s, ...
  |   WaitOne(backoffMs) – prerusenie okamzite pri STOP
  |
  +-- HttpClient.GetAsync(url, ResponseHeadersRead, connectCts[MjpegConnectTimeoutSec])
  |   Timeout: OperationCanceledException → WARN log, continue do vonkajsej slucky
  |
  +-- VNUTORNA SLUCKA (stream read loop)
      |
      +-- stream.ReadAsync(64KB chunk, frameCts[MjpegFrameTimeoutSec])
      |   Timeout: OperationCanceledException → WARN log, break do vonkajsej slucky
      |   bytesRead==0: server zatvoril spojenie → break
      |
      +-- Pridanie dat do 4MB akumulacneho buffera
      |
      +-- JPEG EXTRAKCNA SLUCKA
          |
          +-- MjpegFindSOI(): hlada 0xFF 0xD8 0xFF od zaciatku buffera
          +-- MjpegFindEOI(): hlada 0xFF 0xD9 za SOI
          |
          +-- Ak kompletny JPEG: Cv2.ImDecode(jpegBytes) → Mat
          |   Prvy uspesny frame: totalReconnectAttempts = 0 (reset)
          |
          +-- ProcessFrame(frame) → Pipeline()
          +-- BitmapConverter.ToBitmap() → UI update cez Invoke()

ParameterDefaultCo osetruje

7.5 Pamatova bezpecnost

Predchadzajuca implementacia mala potencialny memory leak: ak new VideoCapture(url) nikdy neskoncila, C++ objekt ostal v pamati natrvalo (GC nevidel unmanaged pamat). Nova implementacia:

  • HttpClient je cisto .NET managed – GC ho spravuje standardne
  • CancellationToken preruší ReadAsync okamzite a deterministicky
  • HttpResponseMessage je ulozeny v _mjpegResponse pod _streamLock a Disposed pri StopCapture() ako doplnkova ochrana
  • Akumulacny buffer (byte[] accBuf, 4MB) je stack-allocated raz na zaciatku a reusovany – ziadna alokacia per-frame

8. Logovanie a MySQL

8.1 Logger (Logger.cs)

Thread-safe asynchronny logger s tymito vlastnostami:

  • Asynchronny zapis – Log() len vlozi do BlockingCollection (neblokuje), background task zapisuje do suboru
  • Rotacia podla dna – log_YYYY-MM-DD.txt
  • Rotacia podla velkosti – Max 5 MB, potom suffix _001, _002...
  • Bounded fronta – Max 10 000 poloziek (ochrana proti memory leak)
  • Log levely – Debug, Info, Warn, Error
  • Format – 2026-02-22 14:30:05.123 | INFO | Sprava

8.2 res.txt (vysledky rozpoznavania)

Kazde vyhodnotenie sa zapise do logs/res.txt vo formate: 2026-02-22 14:30:05<TAB>70.

Rotacia: Ked res.txt presiahne 10 MB, premenuje sa na res_old.txt a zacne novy subor. Stary res_old.txt sa prepise.

8.3 MySQL Writer (MySqlWriter.cs)

Background worker pre zapis do MySQL databazy:

  • Auto-reconnect – pri vypadku spojenia sa automaticky pokusa o opetovne pripojenie
  • Thread-safe fronta – ConcurrentQueue + SemaphoreSlim
  • Auto-create tabulky – CREATE TABLE IF NOT EXISTS pri prvom pripojeni
  • Gating – ak MySqlEnabled = false, Enqueue() neurobi nic

Schema tabulky

CREATE TABLE numreg_readings (
  id          BIGINT NOT NULL AUTO_INCREMENT,

  left_digit  INT NULL,
  right_digit INT NULL,
  text        VARCHAR(16) NULL,
  PRIMARY KEY (id),
  INDEX (ts_utc)
) ENGINE=InnoDB;

Chyba jedneho prikazu neblokuje ostatne – kazdy je obaleny vlastnym try/catch.

9.5 LED ovladanie

Aplikacia moze ovladat LED na ESP32-CAM module cez HTTP GET request: http://<IP>/control?var=led_intensity&val=<0-255>. Slider trkLedIntensity umoznuje nastavit intenzitu v realnom case.

10.2 Thread-safe UI pristup

Capture thread nemoze priamo pristupovat k UI komponentom. Pouzivaju sa helper metody:

  • GetNudValueSafe(NumericUpDown) – cita hodnotu z NumericUpDown, pouziva Invoke ak InvokeRequired
  • GetComboTextSafe(ComboBox) – cita vybrany text z ComboBox, rovnaky princip
  • BeginInvoke(Action) – asynchronny UI update (PictureBox, ListView)

10.3 Mat ownership pattern

OpenCvSharp Mat objekty su nativne (C++ wrapper). Pre bezpecne zdielanie medzi vlaknami sa pouziva pattern:

  1. Pipeline vytvori lokalny Mat (napr. leftPart)
  2. Pod _roiLock sa stary zdielany Mat uvolni (_lastLeft?.Dispose())
  3. Novy Mat sa priradi (_lastLeft = leftPart)
  4. Lokalna premenna sa nastavi na null (aby sa neuvolnila v catch bloku)
  5. UI thread spravi Clone() pod lockom a konvertuje mimo locku

11.3 Stream reconnect

  • Max 50 pokusov bez uspesneho frame → StopCapture()
  • Exponencialny backoff: 1s, 2s, 3s, ... max 30s
  • Reset pri uspechu: kazdy validny frame resetuje pocitadlo

11.4 ListView cap

UI historia (lvNumReg) je limitovana na 500 poloziek. Pri prekroceni sa stara polozka zmaze.

11.5 IDisposable pattern

Vsetky triedy s nativnymi prostriedkami implementuju IDisposable a su korektne uvolnene v OnFormClosing:

  • OnnxDigitRecognizer – InferenceSession
  • Logger – StreamWriter, BlockingCollection, CancellationTokenSource
  • MySqlWriter – MySqlConnection, SemaphoreSlim, CancellationTokenSource
  • VideoCapture – Release + Dispose v StopCapture()
